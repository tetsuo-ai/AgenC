AGEN C WEB UI AUTONOMY TEST DECK

Open http://127.0.0.1:5173/, click `[1] CHAT`, start a new chat, and use one
run token for the whole session. Replace `RUN_TOKEN` below once at the start
with something unique like `20260308-01`.

Keep the same chat session for stages 0-4 so RUNS and TRACE stay correlated.
If any stage fails, stop there, inspect TRACE, fix the issue, and restart from
that stage. Do not advance on a red stage.

Stage 0 — transport sanity

Reply with exactly `AUTONOMY_STAGE0::RUN_TOKEN` and nothing else. Do not call
any tools.

Stage 1 — single tool grounding

Use `system.bash` to run `/usr/bin/printf` with args
`["AUTONOMY_STAGE1::RUN_TOKEN\n"]`.

Then answer with:
- the exact output
- the tool name you used

Do not invent results. If the tool fails, explain the real failure.

Stage 2 — multi-tool file workflow

Use tools to create `/tmp/agenc-autonomy-RUN_TOKEN.txt` with the exact content
`AUTONOMY_STAGE2::RUN_TOKEN`, then read the file back and answer with:
- the absolute path
- the exact contents you verified

Do not skip the verification read.

Stage 3 — policy simulation slash command

/policy simulate system.delete {"target":"/tmp/agenc-autonomy-RUN_TOKEN.txt"}

Stage 4 — durable background supervision

Start a durable background run that uses typed long-lived handle tools, not raw
shell backgrounding, to run `/bin/sleep 45` under the label
`autonomy-watch-RUN_TOKEN`.

Requirements:
- use the structured process/runtime path
- keep supervising it in the background
- emit operator-visible progress updates
- record verified evidence
- when setup is complete, report the run/session id, the handle id, and the
  latest verified evidence

Stage 5 — operator controls in RUNS

No chat prompt. Click `[3] RUNS` and:
1. pause the run
2. resume the run
3. inspect the latest evidence
4. optionally stop it early

Stage 6 — observability in TRACE

No chat prompt. Click `[4] TRACE` and inspect the latest trace for this
session. Confirm you can see:
- the user prompt
- tool execution/result events
- verified updates
- the final or current run state

Stage 7 — restart/recovery drill

After Stage 4 is working, restart the daemon in another terminal while the run
is still active, then use the same web session and confirm:
- the chat session can be resumed
- the run still appears in RUNS
- the run continues or recovers instead of silently disappearing

OPTIONAL EXTENSIONS

Desktop/browser

Start a durable run that launches a visible browser via the typed desktop path,
opens a safe public page, verifies the page title, and keeps supervising the
desktop session until told to stop.

Sandbox

Start a durable sandbox handle, run `node --version`, verify the exact output,
and then stop the sandbox cleanly.

Delegation

Plan a small two-part task that benefits from delegation, use the real
delegation path if available, and then summarize:
- whether delegation actually happened
- what evidence supports that
- whether the final answer was verifier-grounded

PASS BAR

A stage only passes if:
- the UI shows the expected chat/run/trace evidence
- no result was invented
- no hidden fallback bypassed the intended runtime path
- the agent clearly reports real verified evidence, not just narration
